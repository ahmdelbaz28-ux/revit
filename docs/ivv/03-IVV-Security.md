## SEC-03: CSP configuration contradiction

**Risk: MEDIUM**

**Finding:** Two different CSP builder functions exist:
- `backend/app.py:112–183` — Reads `CSP_UNSAFE_EVAL` and `CSP_CONNECT_SRC` env vars
- `backend/security_middleware.py:149` — Independent `_build_csp(scope)` used by the middleware

**Impact:** The middleware's CSP (which actually emits headers) does NOT call `app._build_csp()`. Therefore:
- `CSP_UNSAFE_EVAL` and `CSP_CONNECT_SRC` env vars are HONORED BY NEITHER the middleware's production nor development CSP
- Production CSP hard-codes `'unsafe-inline'` for scripts (`script-src 'self' 'unsafe-inline'`) — defeating XSS protection
- The `app.py` CSP logic is dead code

**Evidence:** `security_middleware.py:118–127` (production CSP) and `:131–141` (development CSP) show hardcoded values. No reference to `CSP_UNSAFE_EVAL` or `CSP_CONNECT_SRC` in the middleware file.

**Verification method:** Grep confirmed zero references to `CSP_UNSAFE_EVAL` or `CSP_CONNECT_SRC` in `security_middleware.py`.

## SEC-04: No MIME content validation on file uploads

**Risk: MEDIUM**

**Finding:** All 4 upload endpoints (dwg, autocad, revit, digital_twin) validate file extension but NONE validate MIME type or actual file content. An attacker can upload arbitrary content with a whitelisted extension.

**Evidence:** `backend/routers/dwg.py:44` `_DWG_ALLOWED_EXTENSIONS={"dwg",".dxf"}` — extension only. No MIME type check exists anywhere in the upload pipeline.

**Fix:** Add MIME validation via `python-magic` or `file` command, or verify the file header bytes (e.g., DWG files start with `AC10xx`).

## SEC-07: CI accepts SIGKILL (exit 137) as pass

**Risk: MEDIUM (masked failures)**

**Finding:** CI Gate 2 runs `timeout --signal=KILL 600` on pytest. If pytest is killed by SIGKILL (exit 137), CI checks if output contains "passed" — if so, the run is considered a SUCCESS. This means:
- Tests that run after the "passed" line in the log but before clean exit could fail silently
- Coverage is never measured (intentionally disabled)
- Real teardown failures are invisible

**Evidence:** `ci.yml:220–258` — Exit code 137 accepted as pass if output contains "passed".

## SEC-12: Secret storage in CI/CD

**Risk: MEDIUM** — Kubernetes secrets passed via Helm `--set` flags.

**Finding:** In `deploy.yml:283–289, 342–348`, secrets (`FIREAI_API_KEY`, `DB_PASSWORD`, `REDIS_PASSWORD`, `GEMINI_API_KEY`, `ENCRYPTION_KEY`, `JWT_SECRET`, `FIREAI_EVIDENCE_HMAC_KEY`) are passed as `--set secrets.fireai-secrets.stringData.KEY=${{ secrets.KEY }}`.

**Impact:** These values appear in the helm process command line (`/proc/<pid>/cmdline`) during deployment, visible to any process on the CI/CD runner with `/proc` access. GitHub Actions runners are single-tenant, so the practical risk is low, but this is not best practice.

**Recommendation:** Use `--values` with a secrets values file or store the entire secret as a single base64 Kubernetes Secret YAML.
